# 计算机和编程最基础概念扫盲

## 一、计算机的“母语”：0和1的故事

首先要明白，计算机本身并不像人，它不能直接理解文字、图片或声音。它的核心是一块小小的芯片——**CPU**（中央处理器），你可以把它叫做计算机的“大脑”。你肯定听过 Intel（英特尔）和 AMD 这些公司，它们就是设计、生产这种芯片的。

那么，CPU 又是怎样工作的呢？

大家还记得初中物理里学过的简单电路吗？一个电池、一个开关、一个灯泡。当开关闭合，电流流通，灯泡就亮了；开关断开，灯泡熄灭。**电路中的开关，只有“通”和“断”两种状态。**

CPU 内部，就密密麻麻地集成了数亿甚至上百亿个肉眼看不见的微型“电子开关”，它们叫**晶体管**。每个晶体管，你同样可以把它看成一个小开关：要么让电流通过（**通**），要么不让电流通过（**断**）。这两种状态，我们人类就习惯用数学符号来表示：

-   **“通”** 我们用 **1** 表示。
-   **“断”** 我们用 **0** 表示。

这样，**一个晶体管就能代表一个数字，要么是0，要么是1。** 这就构成了计算机最底层的语言——**二进制**。

一个开关只能表示两种可能（比如“是”或“否”）。但如果我们把很多个开关组合在一起，能表示的东西就多了。比如，用8个开关一组，就能组合出 $2^8 = 256$ 种不同的状态，足够用来表示我们常用的数字、大小写字母和标点符号了。无数个这样的0和1组合起来，就能表示指令、数据、图像、音乐……世间万物，在计算机里最终都化为了由0和1组成的洪流。

**所以，计算机唯一能“读懂”并直接执行的，就是由0和1组成的、针对特定CPU设计好的命令，这叫“机器语言”。** 对于人来说，看着满屏的0和1比看天书还难，但这确实是计算机的“母语”。

## 二、编程，就是“指挥”与“翻译”

既然计算机只懂0和1，而我们人类习惯用自然语言和数学符号来思考，那怎么让计算机帮我们做事呢？

**编程**，本质上就是我们把想让计算机完成的任务，用一种它“能听懂”的方式告诉它。这整个过程，就像我们指挥一个只懂中文的工匠，但我们却只会说外语。于是，我们需要一个翻译。

在最最早期，计算机科学家们确实就是直接写0和1去指挥机器的，但这效率极低，非常容易出错。于是，人类一直在努力创造更接近我们自己思维方式的“语言”，并同时发明能把这种语言“翻译”成机器语言的工具。这个翻译过程，就是编程的核心。

## 三、编程语言的进化史：越来越像人话

我们来看看人类是怎么一步步把“天书”变成“人话”的。

-   **第一代：机器语言**
    由纯粹的0和1组成，是CPU唯一能直接执行的。难写、难读、难改，只有机器喜欢。

-   **第二代：汇编语言**
    人们开始用一些简短的英文助记符来代替0和1的指令。比如，用 `MOV` 表示移动数据，用 `ADD` 表示加法。这好记多了！但汇编语言仍然与具体的硬件紧密相关，换个CPU可能指令就不一样了。它需要一个叫**汇编器**的程序把它翻译成机器语言。

-   **第三代：高级语言**
    人类追求的是一种更接近于数学公式和自然英语的、与具体硬件无关的语言。这就诞生了高级语言。
    -   **FORTRAN**（1950年代）：第一种广泛使用的高级语言，主要用于科学和工程计算。
    -   **C语言**（1972年诞生）：一个里程碑式的语言。它在贝尔实验室诞生，被用来重写大名鼎鼎的UNIX操作系统。C语言既具有高级语言的易懂性，又能像汇编语言一样直接操作硬件，高效且强大，至今仍是系统编程和嵌入式领域的王者。
    -   **C++**（1980年代）：在C语言的基础上，增加了“面向对象”等更现代的编程思想，让管理大型复杂软件成为可能。它和C语言一脉相承，功能更丰富。
    -   **Java**（1995年诞生）：由Sun公司推出，它的口号是“**一次编写，到处运行**”。Java程序会先被编译成一个叫“字节码”的中间格式，然后在任何装有“Java虚拟机”（JVM）的设备上都能运行。这让它在互联网时代大放异彩。

我们接下来要踏上的，就是C/C++这两门经典编译型语言的学习之路。

## 四、两大家族：编译型与解释型

高级语言写好了，怎么变成机器能懂的0和1呢？这就要靠“翻译官”了。根据翻译方式的不同，高级语言分成了两大家族。

-   **编译型语言（如C、C++）**
    想象你要把一本英文小说送给一位不懂英文的朋友。你会找一位翻译，让他把整本书**一次性全部翻译成中文**，然后装订成一本新书送给朋友。他拿到中文书直接就能流畅阅读。
    在这里，英文原稿就是 **源代码**，翻译官就是 **编译器**，翻译出的中文书就是 **可执行文件**。程序运行时，CPU直接运行那个被编译好的可执行文件，所以运行速度通常很快。

-   **解释型语言（如Python、JavaScript）**
    这次，你请的是一位“同声传译”。你带着朋友去听一场英文演讲，翻译官在旁边，**演讲人说一句，他就立刻翻译一句**给朋友听。演讲结束，朋友也明白了内容，但并没有产生一本完整的“中文演讲稿”。
    在这里，这个“同声传译”就是 **解释器**。它不需要提前生成可执行文件，而是边读源代码边翻译执行。这种方式非常灵活，调试方便，但运行速度通常会慢于编译型。

-   **混合型（如Java）**
    Java则结合了二者。它先用编译器把源代码翻译成“字节码”（不是最终的机器码），然后在运行平台上，由Java虚拟机（JVM）通过解释或即时编译的方式去执行这个字节码，从而实现跨平台。

我们今天要重点了解的，就是**编译器**这个角色。

## 五、编译器：你的专属“翻译官”

**编译器**本身也是一个程序。它的工作，就是把我们写好的高级语言**源代码文件**（比如一个 `.c` 文件），经历复杂的分析、优化，最终生成机器指令。

对现在的你来说，只需要记住两件事：

- **编译器**：把你的源代码翻译成机器语言
- **链接器**（编译器会自动调用）：把翻译好的代码和系统标准功能”打包”，最终生成操作系统可以直接运行的**可执行文件**（Windows 上就是 `.exe` 文件）

市面上有很多优秀的 C/C++ 编译器，我们以后会学习如何通过终端来调遣它们。

## 六、从敲下代码到程序跑起来的全过程

现在，我们把整个过程简化成只有四步。这四步，就是你未来每次写程序都要重复的基本流程。

**第1步：创建源代码文件**

首先，你需要一个文件来存放你写的代码。在电脑上新建一个文件，然后把它的扩展名（就是文件名中 `.` 后面的部分）改成 `.c`（C 源文件），例如 `my_program.c`。这个文件就是一个纯文本文件，你可以用任何文字编辑器打开它。

**第2步：编写源代码**

用电脑上最普通的记事本，打开这个 `my_program.c` 文件，在里面按照 C 语言的语法规则，写下你想让计算机执行的指令。写好后保存。现在，这个文件里装着的，就是你用高级语言编写的“源代码”。

**第3步：编译**

这是最关键的一步。我们会打开一个特殊的“黑窗口”（叫终端），通过输入命令的方式请出编译器。编译器会读取你的 `my_program.c` 源代码文件，检查语法有没有错误，然后把它翻译成计算机能懂的机器语言，并通过链接器把需要的库函数连接进来，最终生成一个**可执行文件**，比如 `my_program.exe`。

这个命令看起来可能类似于：`编译指令 my_program.c -o my_program.exe`。你只需要告诉编译器要翻译哪个源文件、生成的可执行文件叫什么名字就可以了。

**第4步：运行**

现在，你的程序已经“活”了！你可以直接在文件夹里双击 `my_program.exe` 运行它，也可以在终端里输入它的名字（例如 `.\my_program.exe`）来运行。当你按下回车的那一刻，操作系统会将这个 `.exe` 文件加载到内存，CPU开始逐条执行里面的机器指令——你写的程序，就这样在计算机里跑了起来！

在这个过程中，你可能已经发现了，那个“黑窗口”至关重要，我们用它呼唤编译器，也用它运行程序。那到底什么是终端呢？

## 七、终端：与计算机对话的“时光隧道”

想象一下，时间倒退回 50 多年前。那时候的计算机，像一间屋子那么大，而且**根本没有我们现在熟悉的 Windows 桌面、鼠标指针、开始菜单，也没有任何漂亮的图标和窗口**。

那么，那时的程序员是怎么跟这台庞然大物交流的呢？

他们需要一台专门的设备，看起来就像一台老式打字机，又或者是一个只能显示文字的荧幕。这个设备通过长长的线缆连接到远处的主机上。程序员在这个“打字机”上敲下命令，这些命令就传到主机，主机执行完之后，再把结果传回来，显示在这个设备上或打印在纸上。

这个用来输入命令和接收结果的设备，就是**终端**的最初形态。它，就是当时人与计算机交互的唯一桥梁。

后来，个人电脑诞生了，图形化的操作系统（比如 Windows）让一切都变得漂亮和易用。但是，操作系统内部依然保留了一个“回到过去”的入口——一个**黑乎乎的、只有文字的窗口**。在这个窗口里，图形界面能用鼠标完成的几乎所有事，都可以通过输入纯文字的命令来实现。

这个黑窗口，就是现代操作系统中的**终端模拟器**。它在图形界面里，模拟出几十年前那个老式终端的功能。

在 Windows 中，它就叫做“命令提示符”（你可以在开始菜单搜索 `cmd`）或功能更强大的 `PowerShell`。打开它，你会看到一片黑色的背景，上面可能只显示了一行类似 `C:\Users\你的名字>` 的文字和一个闪烁的光标，静静地等待你输入指令。

**为什么学编程，我们要“倒退”回这种原始的方式？**

很多核心的开发工具，比如我们后面要用来翻译代码的编译器、要用来管理版本的 Git，它们最初都是为这种命令行终端设计的。通过终端，我们可以用最直接、最灵活、最强力的方式控制它们。**这正是程序员的“标准工作台”**，很多在图形界面里需要点击十几次的操作，在终端里一句命令就能搞定。

**怎么用它和计算机交流？**
操作其实非常简单，就是**输入指令，然后按回车键。**
你可以现在就试试看：

1.  按键盘上的 `Win` 键（那个四个小方格的键），输入 `cmd`，然后按回车，打开命令提示符。
2.  你会看到光标所在的那一行，提示你当前的位置。
3.  试着输入 `dir`，然后按回车。看到什么了？它列出了一大堆文件和文件夹，对吧？这就是在命令计算机：“给我看看这个目录下都有啥”。
4.  再试试输入 `cd ..`，然后按回车。瞧，你当前的位置变了！这就是在命令计算机：“带我回到上一级文件夹”。

就是这样，你输入一个词（命令），计算机就去执行相应的动作。以后，我们就要在这里，输入类似 `编译器名称 源代码文件` 这样的命令，来把我们的代码变成程序；也要输入 `.\程序名` 来运行我们亲手编写的程序。

终端并不可怕，它只是一条**让工程师能与计算机进行最纯粹、最高效对话的时光隧道**。把它当作你的新伙伴，慢慢熟悉起来就好。

## 八、环境变量：系统的“通讯录”

那么，一个新的问题来了：当我们在终端敲下编译命令时，系统是怎么知道要去哪里找到这个叫做“编译器”的程序的呢？

这就要靠**环境变量**了。环境变量是操作系统里保存的一堆全局设置，就像一本通讯录，记录着重要的信息。其中最重要的一个叫做 **PATH**。

PATH 变量里，记录着一连串文件夹的路径。当你输入一个命令时，系统就会**顺序地**去这些文件夹里寻找同名的可执行程序。一旦找到，就执行它；如果所有路径都找不到，就会提示你”不是内部或外部命令”。

比如，你输入 `notepad` 能打开记事本，就是因为 `notepad.exe` 所在的系统目录（通常为 `C:\Windows\System32`）早就被放在 PATH 里了。

当我们自己安装编译器时，如果安装程序没有自动把编译器的目录添加到 PATH 里，那我们在终端就无法直接调用它。这时，就需要我们**手动配置环境变量**：

1. 在”此电脑”上右键 → 属性 → 高级系统设置 → 环境变量
2. 在系统变量里找到 `Path`，点击编辑
3. 把编译器的可执行文件所在目录添加进去

这样，终端就能在任意地方找到它了。这个操作我们以后会亲手做一遍。

## 九、调试器：程序的“听诊器”和“X光机”

程序可不是写一次就能完美运行的，里面总会有错误，我们称为 **Bug**。光看代码有时很难找出问题，这时就需要**调试器**出马。

调试器也是一个工具，它像一个能让你随时暂停时间的遥控器。你可以：

-   **设置断点**：让程序运行到某一行时自动暂停。
-   **单步执行**：让程序一行一行地慢动作运行，观察每一步的影响。
-   **查看变量**：在暂停时，偷看一眼程序内部某个变量的值，看它是不是你期望的样子。

有了调试器，你就像戴上了一副可以透视程序内部运作的眼镜，任何Bug都无处遁形。我们以后会学习如何使用它，让它成为你最得力的查错助手。

## 十、Git：给你的代码建一座“时光机”

最后，我们来说说两个和写代码密不可分的神器：Git 和 GitHub。

**Git** 是一个**版本控制系统**。它做什么用的呢？

想象你在写一篇非常重要的长篇论文。你可能会这样：`论文_v1.docx`、`论文_v2_修改了开头.docx`、`论文_最终版.docx`、`论文_最终版打死也不改了.docx`…… 很乱对不对？如果哪一天你发现删掉的一段话十分精彩，想找回来，面对一堆文件，就崩溃了。

Git 就是完美解决这个问题的。它像一台**时光机**，能自动、忠实地记录下你的项目文件的每一次改动。

**基本工作流程（都在你的电脑上完成）：**

1.  **初始化仓库**：在你的项目文件夹里，用命令 `git init` 告诉Git：“开始管理这个文件夹吧！” Git会在这里创建一个隐藏的“时光记录本”。
2.  **暂存更改**：你修改了一些代码后，用 `git add 文件名` 命令，把这些改动加入“待提交清单”（暂存区）。相当于告诉Git：“这些改动我准备记录下来了。”
3.  **提交记录**：用 `git commit -m "修好了一个登录的bug"` 命令，正式将暂存区的改动永久记录在仓库里，形成一次“存档”。你写的说明信息（如“修好了一个登录的bug”）让你以后能明白这次改了什么。
4.  **查看历史**：你可以随时用 `git log` 查看所有提交历史，甚至可以神奇地回到任何一个过去的版本。

有了Git，你就可以放心大胆、毫无顾虑地修改你的代码，因为你知道，所有的历史都被安全地保存着，随时可以“读档”。这些操作，我们都会在终端里完成。

## 十一、GitHub：代码的云端家园与社交网络

Git 在你的电脑上建造了一座单机版“时光机”，而 **GitHub** 就是为这座时光机插上网线的**云端托管平台**。

你可以把本地的 Git 仓库，轻松地“推送”（push）到 GitHub 网站上的远程仓库中。这样做的意义在于：

-   **配合远程仓库备份**：Git 默认只把代码和历史保存在本机。将仓库推送到 GitHub、GitLab 等远程平台后，才额外获得异地备份；仅仅执行本地 `git commit` 并不会自动上传到云端。
-   **同步**：你在家里台式机写完代码push，到公司笔记本电脑上可以把它“拉取”（pull）下来继续工作，无缝衔接。
-   **协作**：这是 GitHub 的精髓。世界各地的开发者可以共同为一个项目做贡献，轻松地分享彼此的代码改动。
-   **作品集**：对于程序员来说，GitHub 个人主页就是一份活的技术简历和作品集。

简单说，Git 是工具，GitHub 是为 Git 仓库提供服务的网站。我们以后的练习代码，也都会上传到 GitHub，养成备份和分享的好习惯。
